# Score Draft Task Decomposition Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Build a score-standard-driven task decomposition flow on `/project/:id` that parses regular scoring tables into draft score-follow-up items, lets the bid manager manually assign owners, and only then generates formal project tasks.

**Architecture:** Reuse the existing project detail page and task board. Insert a draft-confirmation layer ahead of formal task creation: backend parses uploaded `.doc/.docx` scoring tables into persisted draft rows, frontend lets the manager review and assign owners, and confirmed rows are transformed into the existing `Task` entity/kanban flow.

**Tech Stack:** Vue 3 + Element Plus + Pinia + Vite, Spring Boot 3.2 + JPA + Flyway, Apache POI for Word table extraction, JUnit 5 + Mockito + MockMvc.

---

## Scope Constraints

- Base the feature on the existing `/project/:id` task decomposition flow in `src/views/Project/Detail.vue`.
- First version only parses explicit and regular scoring tables from `.doc` and `.docx` files.
- System-generated items start as drafts; no formal task is created until the bid manager assigns an owner and confirms.
- Mock mode keeps a deterministic demo path; real parsing requires backend API mode.

## Data Model

### New table: `project_score_drafts`

Columns:
- `id bigint generated by default as identity primary key`
- `project_id bigint not null`
- `source_file_name varchar(255) not null`
- `category varchar(50) not null` (`technical`, `business`, `price`, `other`)
- `score_item_title varchar(255) not null`
- `score_rule_text text not null`
- `score_value_text varchar(100)`
- `task_action varchar(50) not null`
- `generated_task_title varchar(255) not null`
- `generated_task_description text not null`
- `suggested_deliverables text not null` (JSON array string for MVP)
- `assignee_id bigint`
- `assignee_name varchar(100)`
- `due_date timestamp(6)`
- `status varchar(20) not null` (`draft`, `ready`, `skipped`, `generated`)
- `skip_reason varchar(255)`
- `source_page integer`
- `source_table_index integer not null`
- `source_row_index integer not null`
- `generated_task_id bigint`
- `created_at timestamp(6) not null`
- `updated_at timestamp(6) not null`

Indexes:
- `idx_project_score_drafts_project` on `(project_id)`
- `idx_project_score_drafts_status` on `(project_id, status)`

### Formal task linkage

- Reuse existing `tasks` table and `Task` entity.
- Add `generated_task_id` on the draft row instead of creating a separate mapping table.

## API Design

Add these endpoints under the existing `ProjectWorkflowController` base path:

- `POST /api/projects/{projectId}/score-drafts/parse`
  - `multipart/form-data`
  - request fields: `file`
  - behavior: delete previous non-generated drafts for the project, parse regular scoring tables, persist draft rows, return draft list plus summary counts

- `GET /api/projects/{projectId}/score-drafts`
  - behavior: return all score draft rows for the project ordered by category/table/row

- `PATCH /api/projects/{projectId}/score-drafts/{draftId}`
  - behavior: update assignee, due date, title override, description override, status (`draft`/`ready`/`skipped`), skip reason

- `POST /api/projects/{projectId}/score-drafts/generate-tasks`
  - request body: `{ draftIds: number[] }`
  - behavior: validate selected drafts are `ready`, create formal `Task` rows, mark drafts `generated`, return created task summaries

- `DELETE /api/projects/{projectId}/score-drafts`
  - behavior: clear non-generated drafts so the manager can re-parse

## Risks and Decisions

- Existing project document upload only stores metadata, not file bytes. MVP should parse from a dedicated upload inside the score draft dialog rather than relying on already-uploaded documents.
- `.doc` table extraction is more brittle than `.docx`; parser tests need both a happy-path fixture and a conservative fallback path.
- There is no backend user directory endpoint today. MVP should source assignee options from `userStore.users`; a later iteration can replace this with real user lookup.
- Mock mode should remain functional with deterministic draft data even though real document parsing lives in backend API mode.

## Out of Scope

- Parsing PDFs and non-tabular scoring sections
- Automatic owner recommendation
- One scoring row splitting into multiple formal tasks
- Automatic linkage between draft rows and document editor sections
