## 3. 总体判断

### 3.1 已有进展

本轮实现已经完成了第一层闭环搭建，主要包括：

- 前端入口、路由、API 模块和后端 controller 的基础接线
- 告警历史、批量标讯操作、客户商机、AI 质量检查的主要接口补齐
- 前后端可以围绕这 4 个场景形成基本演示链路

### 3.2 尚未达到稳定验收的原因

工程上仍然存在 4 组必须补齐的问题：

1. 数据库迁移缺失
2. 前后端读模型和主读链路仍有不一致
3. Split-First 重构未完成
4. 关键测试闭环不足

## 4. Architecture Review 结论

本阶段共识：**4 个问题，全部选择 A（必须处理）**

### 4.1 数据库迁移缺失

问题：

- 新增了质量检查、质量问题、标讯指派记录等持久化对象
- 但没有与之对应的正式 Flyway migration
- 在 `ddl-auto: validate` 场景下，这会直接影响环境可用性

结论：

- 必须补正式 migration
- 需要同时包含约束、索引和最小必要的回滚思路

### 4.2 标讯读写模型不一致

问题：

- 后端已补批量指派、批量状态更新与 assignment 查询能力
- 但前端列表读模型与后端 canonical status 仍未完全对齐
- 容易出现“写成功但列表展示仍是旧语义”

结论：

- 必须补轻量级标讯列表读模型
- 前端状态显示必须统一通过映射层处理
- 后端 canonical status 不应再被前端旧词汇反向污染

### 4.3 工作台主读链路未完全切到 schedule-overview

问题：

- 后端已补 `GET /api/workbench/schedule-overview`
- 但前端工作台仍未明确完成主读链路切换
- 风险是“接口存在，但页面仍靠旧 store/旧取数路径”

结论：

- 必须把工作台主数据源切到 `schedule-overview`
- 页面、store、API 适配层的职责边界要清楚

### 4.4 客户商机转项目未完成状态回写闭环

问题：

- 后端已有 `convert` 能力
- 但前端主流程仍偏向“跳转去建项目”
- 存在“项目建成了，但预测状态没有回写 converted”的风险

结论：

- 必须把“转项目 -> 回写 prediction 状态/projectId”作为正式闭环补齐
- 不能只保留视觉成功或跳转成功

## 5. Code Quality Review 结论

本阶段共识：**4 个问题，全部选择 A（必须处理）**

### 5.1 标讯状态词汇未统一

问题：

- 前端页面、store、旧逻辑仍混用 `following/contacted/...`
- 后端已引入 canonical status：`PENDING/TRACKING/BIDDED/ABANDONED`
- 双词汇并存会在批量操作、筛选、列表回显中持续制造歧义

结论：

- 必须抽统一的前端状态映射模块
- 页面与 store 都应基于统一映射读写

### 5.2 工作台取数职责不够显式

问题：

- 工作台当前取数路径仍偏隐式
- 业务入口、API 封装、状态容器之间边界不够清晰

结论：

- 必须补显式的 workbench schedule API/adapter
- 页面直接依赖明确的数据入口
- store 只保留状态承载，不再藏业务取数细节

### 5.3 CustomerOpportunityCenter.vue 过大

问题：

- 当前文件已经明显超过 Split-First 约束
- 同时承担容器、筛选、看板、详情、历史抽屉、转项目等多类职责

结论：

- 必须拆分为容器 + 看板 + 客户池/列表 + 详情 + 历史抽屉 + composable
- 本次是刚性整改项，不再继续累积逻辑

### 5.4 AI 质量逻辑已在 useProjectDetailAI.js 聚集

问题：

- 当前虽然文件还未超大，但已经同时承担合规、评分、质量检查与 issue 动作编排
- 如果继续叠加，会快速变成新的职责堆积点

结论：

- 必须把质量检查抽到独立 composable，例如 `useProjectDetailQualityCheck`
- `useProjectDetailAI` 保留编排职责

## 6. Test Review 结论

本阶段共识：**4 个问题，全部选择 A（必须处理）**

### 6.1 AI 文本质量辅助

必须补：

- controller contract tests
- 持久化闭环测试

最少覆盖：

- 发起质量检查
- latest 结果选择
- adopt/ignore 动作
- 无可检查文档时的空态或错误语义
- 返回 DTO 稳定性

### 6.2 客户商机转项目

必须补：

- “转项目后状态回写”闭环测试

最少覆盖：

- prediction 状态更新
- projectId 绑定
- 重复转化语义
- 查询结果与详情结果一致性

### 6.3 标讯批量操作

必须补：

- 批量部分成功/部分失败 contract tests
- 前端提示与列表回刷测试

最少覆盖：

- 批量领取全成功/部分成功/全失败
- 批量指派与 assignment 查询一致
- 批量状态更新的合法/非法迁移
- 单条领取与批量规则一致

### 6.4 工作台与告警

必须补：

- 告警状态迁移 contract tests
- `schedule-overview` 主读链路测试

最少覆盖：

- `acknowledge` 合法/非法迁移
- `resolve` 合法/非法迁移
- 告警历史筛选 contract
- 工作台真实消费 `schedule-overview`
- 告警状态变化后未处理列表/计数刷新

## 7. 强制补项清单

在对外宣称“4 项缺口已完成”之前，必须补齐以下事项：

1. **Flyway migration**
   - 为新增表提供正式迁移脚本

2. **读模型与主读链路一致性**
   - 标讯列表对齐 canonical status 与 assignment read model
   - 工作台切换到 `schedule-overview`
   - 商机转项目完成状态回写

3. **Split-First 重构**
   - 拆分 `CustomerOpportunityCenter.vue`
   - 抽离 `useProjectDetailQualityCheck`
   - 统一标讯状态映射

4. **关键测试补齐**
   - AI 质量 contract
   - 商机转项目闭环
   - 标讯批量部分成功/失败
   - 工作台/告警状态迁移与主读链路

## 8. 推荐执行顺序

建议按以下顺序推进，避免返工：

1. 先补数据库迁移
2. 再补业务闭环
   - 商机转项目回写
   - 工作台主读链路
   - 标讯读模型一致性
3. 再做 Split-First 重构
4. 最后补关键测试并跑整体验证
